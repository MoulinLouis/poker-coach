# UI i18n — design spec

**Date:** 2026-04-21
**Status:** Approved, awaiting plan
**Scope:** Frontend only (React + Vite + TS). No backend changes.

## Goal

Give the web UI a clean French/English translation layer for **static UI chrome**
(navigation, labels, buttons, headings, form copy). Strictly separate from the
existing LLM-advice translation path, which stays untouched.

In this spec, *UI chrome* means any human-readable string literal written inside
JSX or component code — even when it maps from a backend/domain value at the
render site (e.g. the `"Reg"` / `"Unk."` labels shown for the `reg` /
`unknown` villain profiles). The distinction is **where the string is written**
(frontend source) vs. **where it originates at runtime** (backend response).
Only the former is in scope.

## Non-goals

- Backend-originated strings (error messages, engine values, preset IDs) stay in English.
- Interpolation (`{name}`-style placeholders). Compose at call sites if needed; add later if a real need appears.
- Pluralization. Not used today; use a one-off `Intl.PluralRules` call if/when required.
- Locale-aware number/date formatting.
- Additional locales beyond EN and FR.

## Constraints / context

- App is small: ~4,200 LOC across ~20 components. All UI text currently hardcoded English.
- Brand has French flavor (`Maison du Coup`), so French coverage is a product-fit feature, not just an accessibility add-on.
- Existing `frontend/src/api/useTranslation.ts` is a per-blob LLM-output translator hitting `/api/translate` (Haiku 4.5). It is **not** i18n and must not be conflated with the new system. It will be renamed to `useAdviceTranslation` in this work for clarity (resolves the name collision with any future `useTranslation`-named hook and makes intent obvious at call sites). Rename touches three files: the hook module, its test, and `AdvicePanel.tsx` (the sole consumer, two call sites).

## Architecture

A homegrown tiny i18n module — no library dependency. Rationale: only two
locales, bounded string set, TypeScript already provides the strong guarantees
that libraries like `react-i18next` otherwise carry (exhaustive-key checking,
missing-translation detection at compile time). Libraries would add ~40 KB
gzipped and plugin surface for zero net benefit at this scale.

```
frontend/src/i18n/
├── index.ts           # re-exports LocaleProvider, useLocale
├── LocaleContext.tsx  # React context + provider + useLocale hook
├── dictionaries.ts    # type Dict = typeof en; exports { en, fr }
├── en.ts              # source of truth — EN strings
└── fr.ts              # FR strings, typed as `typeof en`
```

### Public API

```ts
// LocaleContext.tsx
export type Lang = "en" | "fr";
export function LocaleProvider({ children }: { children: ReactNode }): JSX.Element;
export function useLocale(): {
  lang: Lang;
  setLang: (l: Lang) => void;
  t: (key: DictKey) => string;
};
```

- `DictKey` is a compile-time union of dotted paths into the dictionary
  (e.g. `"nav.live"`, `"setup.effectiveStack"`). Typos fail `tsc`.
- `t()` walks the key path. If the key does not resolve at runtime (should be
  impossible under TS), returns the key string and `console.warn`s. Prevents
  blank UI.

### Dictionary shape

Nested by **surface**, not by semantic grouping — one top-level key per
component/route:

if any key is missing or extra.

### Persistence

- On `LocaleProvider` mount: read `localStorage.getItem("locale")`. If
  `"en"` or `"fr"`, use it; otherwise default to `"en"`.
- `setLang(l)` writes `localStorage.setItem("locale", l)` and updates state.
- No browser-language auto-detection (per design decision — predictable default).

## Language switcher

New component `frontend/src/components/LocaleToggle.tsx`. Two text buttons
`EN | FR` with a gold hair separator. Styled to match the existing header
metadata cluster (mono caps, `tracking-[0.28em]`, parchment-dim / bone-white
for inactive / active).

Placement: header right cluster in `App.tsx`, adjacent to `Heads-Up · No Limit`
and `v.0427`. **Always visible**, including mobile — pulled out of the
`hidden md:flex` group since it's functional, not decorative.

`data-testid="locale-en"` and `data-testid="locale-fr"` for E2E targeting.

## Separation from LLM-advice translation

Two distinct paths, must not cross-contaminate:

| Layer | What it translates | How |
|-------|-------------------|-----|
| **i18n (this spec)** | Static UI chrome written in JSX | Dictionary lookup, synchronous |
| **LLM-advice translation** (existing) | Runtime LLM output (reasoning blobs) | Backend Haiku 4.5 call, async, per-text |

In `AdvicePanel.tsx`, both coexist: `t("advice.coach")` for the header label,
`useAdviceTranslation(stream.reasoning)` for the coach's reasoning text.

## Migration order

1. **Skeleton commit** — add `i18n/` module, provider, toggle, seed dictionary
   with only `nav.*` keys. Wire provider in `main.tsx`, toggle in `App.tsx`.
   The app ships as partially bilingual at this point: nav labels respond to
   the toggle, everything else stays English until its surface lands in a
   later commit. Rename `useTranslation` → `useAdviceTranslation` here (not at
   the end) so no one imports the old name during migration.
2. **One surface per commit**, smallest first:
   - `App.tsx` (nav + brand area — `Maison du Coup` stays literal as a brand term)
   - `SetupPanel`
   - `AdvicePanel` (chrome only — reasoning text still flows through `useAdviceTranslation`)
   - `CostFooter`, `PokerTable`, `CardPicker`, `BoardPicker`, `ActionBar`, `HandSummary`, `Seat`, `Pot`, `Board`
   - Routes: `LiveCoach`, `SpotAnalysis`, `History`, `Prompts`

## Testing

- `frontend/src/i18n/LocaleContext.test.tsx` (vitest):
  - Provider defaults to `"en"` when `localStorage` is empty / invalid.
  - Provider reads existing `localStorage.getItem("locale")`.
  - `setLang` updates state and writes `localStorage`.
  - `t("nav.live")` returns EN string under `lang="en"`, FR under `lang="fr"`.
  - Missing-key path (if synthesized via a cast) returns the key and emits `console.warn`.
- Existing component tests that assert on English text (e.g. `getByText("New hand")`)
  remain valid — EN is the default in a fresh test environment.
- Playwright smoke: toggle to FR on `LiveCoach`, assert a known FR string
  appears; toggle back, assert EN returns. Reload page, assert FR persisted.
- Type-level completeness is enforced by `fr: typeof en` — no runtime test needed for parity.

## Error handling

- `localStorage` unavailable (private mode, disabled): `setLang` silently skips
  the write; in-memory state still works for the session.
- Missing key at runtime (should be unreachable): return key, `console.warn` in dev.
- No network, no backend, no async — zero failure modes outside of browser storage.

## What this spec does **not** change

- `/api/translate` endpoint and `backend/src/poker_coach/translation.py`.
- Prompt/LLM behavior.
- Test infrastructure or E2E harness.
- Any backend error or response strings.
